## 项目主要需求

### 项目概述

CommunityGov 是一款社区治理与便民服务平台的后端 API 服务，面向社区居民、网格员、部门管理员和超级管理员四类角色，提供投诉工单管理、便民服务申请、社区公告发布和网格员管理等核心功能。系统采用 NestJS + TypeORM 构建，PostgreSQL 持久化数据，Redis 缓存会话和热点数据，JWT 实现认证授权。

### 核心实体

#### 1. 投诉工单 Complaint

```typescript
interface Complaint {
  id: number;
  ticketNo: string;          // 工单编号，格式：TS-YYYYMMDD-XXXX
  reporterId: number;        // 投诉人ID
  category: ComplaintCategory; // 物业/环境/安全/噪音/设施/其他
  title: string;             // 投诉标题
  description: string;       // 详细描述
  images: string[];          // 图片URL列表
  location: string;          // 地点描述
  priority: Priority;        // LOW/MEDIUM/HIGH/URGENT
  status: TicketStatus;      // 工单状态（流转状态机）
  assignedDeptId: number | null; // 分配的部门ID
  handlerId: number | null;  // 处理人ID（网格员或部门人员）
  resolvedAt: Date | null;   // 解决时间
  closedAt: Date | null;     // 关闭时间
  createdAt: Date;
  updatedAt: Date;
}
```

状态流转状态机：
```
SUBMITTED → ASSIGNED → IN_PROGRESS → PENDING_REVIEW → RESOLVED → CLOSED
                                    ↗                ↗
                              REOPENED ─────────────
```

业务动作：
- **提交**（居民）：填写分类、标题、描述、图片、地点，系统自动生成工单编号，初始状态为 SUBMITTED。
- **分派**（部门管理员）：指定处理部门和处理人，状态变更为 ASSIGNED。URGENT 优先级工单必须在 1 小时内分派。
- **处理**（网格员/部门人员）：开始处理时状态变为 IN_PROGRESS，处理完成提交审核时变为 PENDING_REVIEW。
- **完结**（部门管理员）：审核通过后状态变为 RESOLVED，记录解决时间；居民确认后变为 CLOSED。
- **重开**（居民/管理员）：对已解决的工单不满意可重开，状态变为 REOPENED。

关联表：
- `complaint_logs`：工单流转日志（每次状态变更记录操作人、时间、备注、变更前后状态）
- `complaint_comments`：工单评论（处理人员和居民之间的沟通记录）

#### 2. 便民服务 ServiceRequest

```typescript
interface ServiceRequest {
  id: number;
  requestNo: string;         // 申请编号
  type: ServiceType;         // 证明开具/信息查询/政策咨询/代办服务
  requesterId: number;       // 申请人ID
  title: string;             // 申请标题
  description: string;       // 详细描述
  attachments: string[];     // 附件URL列表
  status: 'PENDING' | 'ACCEPTED' | 'IN_PROGRESS' | 'COMPLETED' | 'REJECTED';
  handlerId: number | null;  // 受理人ID
  dueDate: Date | null;      // 承诺办结日期
  completedAt: Date | null;  // 实际办结时间
  rejectionReason: string | null; // 驳回原因
  rating: number | null;     // 申请人评价(1-5)
  ratingComment: string | null; // 评价内容
  createdAt: Date;
  updatedAt: Date;
}
```

业务动作：
- **申请**（居民）：选择服务类型，填写标题和描述，提交后系统生成申请编号。
- **受理**（网格员/部门人员）：受理申请，设定承诺办结日期，状态变更为 ACCEPTED。
- **办结**（处理人）：完成办理后标记为 COMPLETED，记录实际办结时间。
- **驳回**（处理人）：不符合办理条件时驳回，填写驳回原因。
- **评价**（居民）：办结后居民可对服务进行评分和评论。

### 横切关注点

#### 1. 认证授权 + RBAC

**角色定义**：
- `RESIDENT`（居民）：提交投诉、申请服务、查看公告、评价服务
- `GRID_WORKER`（网格员）：处理工单、受理服务、查看区域数据
- `DEPT_ADMIN`（部门管理员）：分派工单、审核完结、管理网格员、查看部门统计
- `SUPER_ADMIN`（超级管理员）：审核公告、全局管理、查看全平台统计

**实现要求**：
- JWT 双 Token 机制：accessToken（15分钟过期）+ refreshToken（7天过期）
- `auth.middleware.ts`：验证 JWT Token，解析用户信息注入 request
- `role.middleware.ts`：基于角色的权限守卫，装饰器方式声明所需角色
- **数据范围过滤**：在 Service 层根据角色过滤查询范围（居民只看自己的，网格员看分配的，部门管理员看本部门的）
- 密码使用 bcrypt 加密，salt rounds = 10

**触达范围**：DB（users 表、roles 权限表）→ 中间件（auth、role）→ 路由守卫 → Service 层数据过滤

#### 2. 操作日志 + 工单流转日志

**操作日志（AuditLog）**：
- 通过 `audit.middleware.ts` 拦截所有写操作（POST/PUT/DELETE）
- 记录操作人、操作类型、操作实体、实体ID、变更详情（before/after diff）、IP、UA
- 使用 TypeORM subscriber 或 middleware 方式实现，不侵入业务代码
- audit.service.ts 提供 `log()` 方法和查询接口

**工单流转日志（ComplaintLog）**：
- 每次工单状态变更时自动记录
- 字段：`{ complaintId, fromStatus, toStatus, operatorId, comment, createdAt }`
- 在 complaints.service.ts 的状态流转方法中调用

**触达范围**：DB（audit_logs、complaint_logs 表）→ 中间件（audit）→ Service 层（状态流转方法）

### 全局异常处理

后端统一异常中间件 `error-handler.middleware.ts`，覆盖所有异常场景：

- **业务异常**（BusinessException）：自定义异常类，携带 HTTP 状态码和业务错误码，如 `COMPLAINT_NOT_FOUND`、`INVALID_STATUS_TRANSITION`、`PERMISSION_DENIED`。
- **参数校验异常**（ValidationException）：DTO 校验失败时自动拦截，返回 400 + 详细字段错误信息。
- **认证异常**（UnauthorizedException）：Token 无效或过期时返回 401 + 清晰错误提示。
- **权限异常**（ForbiddenException）：角色权限不足时返回 403。
- **未知异常**：捕获所有未处理异常，返回 500 + 通用错误信息，同时记录到日志文件。

异常响应统一格式：
```json
{
  "code": 40001,
  "message": "工单不存在",
  "error": "COMPLAINT_NOT_FOUND",
  "timestamp": "2026-01-15T10:30:00Z",
  "path": "/api/complaints/123"
}
```

### 代码严禁合并职责到单一文件

- 每个路由文件（routes/）只负责路由注册和中间件绑定，不包含业务逻辑。
- 每个控制器（controllers/）只负责参数解析和响应格式化，核心逻辑在 service 层。
- 每个 Service 只管理一个实体的业务逻辑，跨实体操作通过 Service 间调用实现。
- Entity 文件只定义数据模型和映射关系，不包含业务方法。
- 中间件（middlewares/）只负责横切关注点（认证、日志、异常处理），不包含业务逻辑。
- 配置文件（config/）集中管理数据库、Redis、JWT 等配置，不在业务代码中硬编码。
- 数据库迁移和种子数据放在 `database/` 目录，与业务代码完全分离。

### 核心实体数据流

每个核心实体必须贯穿完整数据流：

```
DB (PostgreSQL) → Entity(models/) → Service(services/) → Controller(controllers/) → Route(routes/) → API 响应
                                             ↑
                                    中间件(middlewares/)     枚举常量(constants/)
```

以 Complaint 投诉工单为例：
1. `src/constants/enums.ts` 定义 `ComplaintCategory`、`TicketStatus`、`Priority`
2. `src/models/complaint.entity.ts` 定义 Complaint Entity，引用枚举作为列类型
3. `src/database/migrations/002-create-complaints.ts` 创建 complaints 表
4. `src/services/complaints.service.ts` 实现工单的 CRUD、状态流转、统计查询，调用 `audit.service.ts` 记录日志
5. `src/controllers/complaints.controller.ts` 解析请求参数，调用 Service，格式化响应
6. `src/routes/complaints.routes.ts` 注册路由，绑定 `auth.middleware.ts` 和 `role.middleware.ts`
7. `src/middlewares/audit.middleware.ts` 拦截写操作，调用 `audit.service.ts` 记录审计日志

## 代码存放规则

这是前后端分离项目。

前端代码放在 `frontend/`：
页面，组件，路由,状态管理,API 请求,类型定义,样式和资源等放在对应的子文件夹

后端代码放在 `backend/`：
数据库脚本放在 `database/`。

## 部署要求

项目必须支持 Docker Compose 一键部署，且在 **任意目录名（包括中文目录名）** 下都能正常启动。

本项目为纯后端服务，Docker Compose 仅包含后端和数据库服务（无前端容器）。

需要提供以下文件：
- 根目录 `docker-compose.yml`：编排后端、PostgreSQL、Redis
- 根目录 `.env`：包含 `COMPOSE_PROJECT_NAME=wjegov`，确保 Compose 项目名不依赖目录名
- `backend/Dockerfile`：后端项目构建与运行镜像（多阶段构建）
- `.env.example`：所有环境变量的示例文件，附注释说明

docker-compose.yml 要求：
- **不要写 `version:` 字段**
- 顶层声明 `name: wjegov`
- 所有服务使用 `container_name` 命名容器，带 `${COMPOSE_PROJECT_NAME:-wjegov}` 前缀：
  - `container_name: ${COMPOSE_PROJECT_NAME:-wjegov}-backend`
  - `container_name: ${COMPOSE_PROJECT_NAME:-wjegov}-postgres`
  - `container_name: ${COMPOSE_PROJECT_NAME:-wjegov}-redis`
- PostgreSQL 数据使用命名卷持久化
- PostgreSQL 配置 `healthcheck`（使用 `pg_isready`）
- 后端 `depends_on` PostgreSQL 健康和 Redis
- Redis 配置 `healthcheck`（使用 `redis-cli ping`）
- 后端暴露端口 38306
- 所有可变配置通过环境变量注入

```yaml
# docker-compose.yml 示例结构（仅供参考，需根据实际需求完善）

services:
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    container_name: ${COMPOSE_PROJECT_NAME:-wjegov}-backend
    ports:
      - "38306:38306"
    environment:
      - DATABASE_HOST=postgres
      - DATABASE_PORT=5432
      - DATABASE_USER=${DB_USER:-postgres}
      - DATABASE_PASSWORD=${DB_PASSWORD:-postgres}
      - DATABASE_NAME=${DB_NAME:-wjegov}
      - REDIS_HOST=redis
      - REDIS_PORT=6379
      - JWT_SECRET=${JWT_SECRET:-your-secret-key}
      - JWT_EXPIRES_IN=${JWT_EXPIRES_IN:-15m}
      - JWT_REFRESH_EXPIRES_IN=${JWT_REFRESH_EXPIRES_IN:-7d}
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_started
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:38306/api/health"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

  postgres:
    image: postgres:16-alpine
    container_name: ${COMPOSE_PROJECT_NAME:-wjegov}-postgres
    environment:
      POSTGRES_USER: ${DB_USER:-postgres}
      POSTGRES_PASSWORD: ${DB_PASSWORD:-postgres}
      POSTGRES_DB: ${DB_NAME:-wjegov}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER:-postgres}"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    container_name: ${COMPOSE_PROJECT_NAME:-wjegov}-redis
    volumes:
      - redis_data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

volumes:
  postgres_data:
  redis_data:
```

示例 curl 命令（README 中需包含）：

```bash
# 1. 注册居民账号
curl -X POST http://localhost:38306/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"resident1","password":"Pass1234","realName":"张三","phone":"13800138000"}'

# 2. 登录
curl -X POST http://localhost:38306/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"resident1","password":"Pass1234"}'

# 3. 提交投诉（使用登录返回的 accessToken）
curl -X POST http://localhost:38306/api/complaints \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <accessToken>" \
  -d '{"category":"NOISE","title":"楼下装修噪音扰民","description":"连续三天早上7点开始电钻","priority":"HIGH","location":"3号楼2单元"}'

# 4. 查询工单列表（分页+筛选）
curl -X GET "http://localhost:38306/api/complaints?page=1&limit=10&status=SUBMITTED" \
  -H "Authorization: Bearer <accessToken>"

# 5. 分派工单（需部门管理员权限）
curl -X PUT http://localhost:38306/api/complaints/1/assign \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <adminToken>" \
  -d '{"handlerId":5,"assignedDeptId":2,"comment":"转交物业部门处理"}'
```
